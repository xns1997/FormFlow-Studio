# FormFlow 项目创建与交付规范

这份文档用于把 FormFlow Studio 现有能力整理成一套可执行、可验收、可交接的标准流程。

目标有三件事：

1. 统一“创建项目 → 导入数据表 → 设计表单 → 绑定事件/流程 → 配置项目行为 → 测试 → 转入使用模式”的操作顺序。
2. 明确每个阶段的输入、输出、完成标准、禁忌和职责边界。
3. 留下一套其他智能体可直接遵循的项目编排规范。

---

## 1. 当前系统事实基线

基于当前仓库实现，系统已经具备以下能力：

- 项目包结构：`project.json`、`forms/`、`data/`、`workflows/`、`global-behaviors.json`、`outputs/`、`release.json`
- 项目创建方式：空白、模板、ZIP 导入
- 数据导入与配置：支持 Excel / CSV / JSON，支持 sheet 级 key 配置、冻结区、筛选、列宽等
- 数据准备：服务端搜索、筛选、排序、分页、Key 定位、完整结果导出，以及带版本冲突保护的跨页批量写回
- 表单设计：从数据生成五种用途表单、字段拖入与补齐、任务式属性编辑、预览模式、流程触发器、控件事件脚本
- 流程编排：工作流画布、参数映射、216 个节点包、10 个高阶宏、5 个流程配方、节点执行与输出预览
- 行为能力：全局行为、工作表行为、表单行为、受控规则 DSL、行为模板、方法库、行为测试、事件参考文档
- 测试运行：表单渲染、行切换、事件执行、流程联动、导出、变更日志
- 发布与使用模式：`release.json` 保存 `design | test | use` 状态、默认表单/Sheet 和设计入口权限；独立使用模式隐藏创作入口
- 质量门禁：开发任务总览统一检查字段绑定、按钮动作、流程引用、联动冲突、主键与自动生成测试
- 智能体交付：仓库内置 `formflow-project-editor` skill，支持项目 inspect/create/normalize/validate/pack/unpack

当前系统仍存在的主要增强项是：

1. 自动生成的测试用例、Mock seed、隔离夹具和最近运行摘要已通过 `testing/testing.json` 持久化，并可由项目智能体回归执行。
2. 发布门禁已有交互界面和项目智能体预检，但还缺少独立报告导出和可供 CI 直接消费的项目级结果文件。
3. 使用模式已有入口隔离与 release 权限位，但还缺少设计者、测试者、使用者、管理员的细粒度运行态角色策略。
4. 工作流适用性尚未根据脚本复杂度主动提示重构为流程。

因此，下面的规范以当前实现为基线，并把尚未补齐的能力单独标明。

## 2. 核心原则

### 2.1 一个项目必须先定义数据边界，再设计表单

禁止先随意拖控件、后补数据主键和 sheet 结构。否则测试和回写都不稳定。

### 2.2 事件不是流程，流程也不是行为

- 事件：触发时机
- 行为：轻量规则、脚本、表单内联动
- 流程：跨步骤、可复用、可观测、可调试的编排

判断原则：

- 同表单内、一步完成、只改几个字段：优先行为
- 涉及查询、计算、写回、导出、跨控件复用：优先流程

### 2.3 行为必须分层

行为分为三层：

- 全局行为：跨项目表单共享的规则
- 工作表行为：绑定到某个数据表 / sheet 的规则
- 表单行为：绑定到具体表单和控件交互的规则

当前实现通过 `sheetBehaviors` 与 `data/<table-id>.behaviors.json` 持久化工作表行为；编辑与执行时必须同时使用 `tableId + sheetName` 定位作用域。

### 2.4 测试不是最后补一下

每个阶段都要留下可验证产物，最终测试只是总验收，不是第一次发现问题。

### 2.5 使用模式必须和设计模式隔离

项目进入使用模式后，默认不再暴露设计器、行为编辑器、流程编辑器，只保留运行、提交、导出和审计入口。

## 3. 标准八阶段流程

## 阶段 1：创建项目

### 输入

- 项目名称
- 项目用途
- 创建方式：空白 / 模板 / ZIP 导入

### 操作规范

1. 先确定项目类型：
   - 数据录入型
   - 查询编辑型
   - 审批型
   - 批处理型
   - 分析预测型
2. 再选择项目骨架：
   - 空白项目：适合强定制
   - 模板项目：适合快速起步
   - ZIP 导入：适合复用旧项目
3. 创建后立即补齐元信息：
   - 名称
   - 描述
   - 作者
   - 标签
   - 版本号

### 输出物

- `project.json`
- 初始项目设置
- 初始表单或模板骨架

### 完成标准

- 项目 ID 稳定
- 基础信息齐全
- 已明确默认业务场景

### 禁忌

- 不允许“先建项目，后决定是录入还是审批”
- 不允许名称、数据用途、写回策略都为空

---

## 阶段 2：导入数据表

### 输入

- Excel / CSV / JSON 数据源

### 操作规范

1. 为每个数据源确认：
   - 文件类型
   - 数据用途
   - 是否可回写
2. 为每个 sheet 配置：
   - key 字段，支持组合 key
   - 表头高度、行高
   - 冻结行列
   - 筛选 / 排序 / 自动列宽
3. 明确数据角色：
   - 主业务表
   - 查询参考表
   - 枚举字典表
   - 审计 / 历史表

### 输出物

- `data/_index.json`
- `<tableId>.meta.json`
- 每个 sheet 的 key 配置和显示配置

### 完成标准

- 每个会参与编辑或回写的 sheet 都有 key
- key 校验通过，无空值或重复
- 已标记哪些表只读、哪些表可写

### 禁忌

- 未定义 key 就进入编辑类表单设计
- 把参考表和主写回表混成同一种角色

---

## 阶段 3：设计表单

### 输入

- 已稳定的数据表结构
- 业务流程草图

### 操作规范

1. 先定表单职责，而不是先堆控件：
   - 新建表单
   - 查询表单
   - 编辑表单
   - 审批表单
   - 明细查看表单
2. 为每个表单定义 `formMode`：
   - `create`
   - `edit`
   - `detail`
   - `lookup-edit`
3. 表单内控件必须分组：
   - 主键/定位区
   - 业务输入区
   - 结果展示区
   - 操作按钮区
4. 每个可写控件都要明确：
   - 字段名
   - 数据来源
   - 是否必填
   - 是否只读
   - 默认值策略

### 输出物

- `forms/<form-id>.json`

### 完成标准

- 表单结构完整
- 控件命名可读且稳定
- 每个按钮都有清晰职责
- 可从预览模式完成一次主操作路径

### 禁忌

- 只写 label，不写稳定的 `name`
- 在一个按钮里混合“查询 + 保存 + 导出”

---

## 阶段 4：设计控件点击事件

### 输入

- 已完成的表单结构

### 操作规范

1. 先做轻量事件：
   - `onChange`
   - `onBlur`
   - `onFocus`
   - `onClick`
   - `onTabChange`
2. 轻量事件只负责：
   - 表单内字段联动
   - UI 状态切换
   - 参数准备
   - 简单校验提示
3. 事件代码必须避免直接写复杂业务逻辑。

### 推荐判断

- 事件代码超过 20 到 30 行，优先改为流程
- 同一逻辑被两个按钮复用，优先改为流程
- 需要数据查询、节点计算、写回、导出，优先改为流程

### 输出物

- 控件 `props.events`
- 控件 `props.flowTriggers`
- 必要时的 `linkageRules`

### 完成标准

- 事件职责单一
- 没有把复杂业务塞进按钮脚本
- 每个事件都能说明“触发时机、输入、输出”

---

## 阶段 5：能用流程就用流程，并绑定到控件事件

### 输入

- 已识别出的复杂交互需求

### 操作规范

1. 优先抽成流程的场景：
   - 数据查询
   - 多步计算
   - 条件分支
   - 批量处理
   - 写回与导出
   - 需复用的业务动作
2. 工作流必须声明：
   - 名称
   - 业务意图
   - 输入参数
   - 输出结果
   - 是否允许中途终止
3. 控件事件只负责：
   - 收集参数
   - 调用目标流程
   - 处理成功/失败反馈

### 流程设计标准

- 一个流程只服务一个业务目标
- 节点命名必须可读
- 输入变量命名与表单字段命名一致
- 输出必须能被测试页或事件日志看见

### 输出物

- `workflows/workflows.json`
- 表单事件到流程的参数映射

### 完成标准

- 所有复杂业务操作都已流程化
- 控件事件和流程分工清晰
- 参数映射无需猜测

### 禁忌

- 在脚本里手写一段本可由流程表达的多步逻辑
- 一个流程同时承担“查询、审批、导出、清空表单”四种目标

---

## 阶段 6：编辑项目行为

这是本规范最重要的补强部分。

### 6.1 目标作用域

项目行为必须分成三层：

1. 全局行为
2. 工作表行为
3. 表单行为

### 6.2 全局行为

适用场景：

- 项目加载初始化
- 提交前统一校验
- 提交成功 / 失败统一提示
- 全项目通用安全限制

当前存储建议：

- `global-behaviors.json`

### 6.3 工作表行为

适用场景：

- 某个 sheet 的行装载规则
- 某个 sheet 的 key 校验规则
- 某个 sheet 的默认值、映射值、字典联动
- 某个 sheet 的写回前清洗

目标存储建议：

- `sheet-behaviors/<tableId>.<sheetName>.json`
  或
- `data/<tableId>.behaviors.json`

当前现状说明：

- 仓库内尚未看到独立的工作表行为持久化结构
- 因此这是首个明确建议补齐的一等能力

### 6.4 表单行为

适用场景：

- 表单加载默认值
- 字段联动
- 按钮前置校验
- 标签页切换后的界面更新

当前存储建议：

- `forms/<form-id>.behaviors.json`

### 6.5 分层决策表

| 需求 | 放在哪一层 |
|------|------------|
| 所有表单提交前都要做手机号校验 | 全局行为 |
| 员工表的 ID、部门编码要统一补齐 | 工作表行为 |
| “审批意见”只有在选择“拒绝”时才显示 | 表单行为 |

### 6.6 完成标准

- 每条行为都有明确作用域
- 不再把 sheet 规则混进表单脚本
- 不再把控件联动误写成全局行为

---

## 阶段 7：测试

### 测试分四层

1. 结构测试
2. 事件测试
3. 流程测试
4. 业务验收测试

### 7.1 结构测试

检查：

- 缺失绑定
- 重复 key
- 无效流程引用
- 死循环风险
- 不可达按钮

### 7.2 事件测试

检查：

- `onChange` 是否引发正确联动
- `onBlur` 是否给出正确校验
- `onClick` 是否能成功触发流程

### 7.3 流程测试

检查：

- 参数映射是否正确
- 节点输出是否稳定
- 错误分支是否可观测
- 写回/导出是否符合预期

### 7.4 业务验收测试

至少覆盖以下用例：

- 主路径成功
- 必填缺失
- key 冲突
- 查无数据
- 多条匹配
- 提交失败
- 只读模式
- 回写关闭模式

### 输出物

- 测试记录
- 变更日志样本
- 失败案例与修复记录

### 完成标准

- 每个表单至少 1 条主路径用例
- 每个关键流程至少 1 条成功用例和 1 条失败用例
- 所有允许写回的项目都验证过回写行为

---

## 阶段 8：转变为使用模式

这是当前项目包已支持的发布状态，不应只理解为“用户开始用了”。

### 使用模式定义

使用模式 = 面向最终业务用户的运行态，而不是创作态。

### 进入条件

- 数据源角色清楚
- 所有关键 sheet 已有 key
- 主表单已指定
- 事件和流程已绑定完成
- 行为分层已完成
- 测试通过
- 发布策略已确认

### 使用模式最低要求

1. 默认入口是运行/表单页，而不是编辑器
2. 非管理员默认看不到设计、行为、流程画布
3. 是否允许写回由项目发布设置统一控制
4. 运行态需要保留：
   - 表单操作
   - 提交
   - 导出
   - 审计日志
   - 错误提示

### 当前实现与后续增强

- 已实现：项目级 `mode: design | test | use`
- 已实现：默认表单、默认 Sheet 与设计器/行为/流程入口权限位
- 已实现：独立使用模式入口，默认不展示创作界面
- 待增强：按角色控制进入和回退权限，而不只依赖项目级布尔权限位
- 待增强：可配置的运行时首页与管理员解锁审计

## 4. 阶段门禁表

| 阶段 | 必须满足 | 不满足时禁止进入 |
|------|----------|------------------|
| 创建项目 | 项目元信息已填写 | 导入数据 |
| 导入数据表 | 关键 sheet 已设 key | 编辑类表单设计 |
| 设计表单 | 控件命名和表单职责清楚 | 大规模事件脚本编写 |
| 设计事件 | 轻量事件已拆清 | 复杂业务全塞脚本 |
| 建流程并绑定 | 流程输入输出明确 | 正式测试 |
| 配置项目行为 | 全局/工作表/表单作用域明确 | 转使用模式 |
| 测试 | 主路径和失败路径已覆盖 | 对外使用 |
| 使用模式 | 发布策略、权限、入口已定 | 无 |

## 6. 功能查漏补缺清单

以下状态基于 2026-07-14 的仓库实现复核。

### P0

1. ✅ 工作表行为一等模型、存储结构和项目包读写已实现
2. ✅ 项目级 `design | test | use` release 状态和独立使用入口已实现
3. ✅ 发布门禁已检查字段、按钮、流程引用、主键和自动测试

### P1

4. ✅ `defaultFormId` / `defaultSheet` 已进入 `release.json`
5. ✅ 项目级开发总览已汇总：
   - key 是否齐全
   - 表单是否有未绑定字段
   - 是否存在失效流程引用
   - 是否存在未测试的关键流程
6. ⏳ 增加工作流适用性提示：
   - 当控件事件脚本过长时，提示“建议改为流程”

### P2

7. ✅ 增加测试用例持久化和回归测试入口
8. ⏳ 增加“发布前检查”报告导出
9. ⏳ 增加运行态权限分级：设计者 / 测试者 / 使用者 / 管理员

## 7. 给其他智能体的执行规范

当其他智能体接手一个 FormFlow 项目时，必须按下面顺序工作。

### 7.1 强制流程

1. 先读项目结构和 `project.json`
2. 再读 `data/`，确认主表、参考表、key、可写表
3. 再读 `forms/`，确认表单职责和 `formMode`
4. 再读 `workflows/`，确认复杂业务是否流程化
5. 再读全局 / 表单行为，判断是否存在作用域混乱
6. 最后才修改事件、流程或表单

### 7.2 强制判断

- 如果逻辑是跨表、跨步骤、可复用，优先改流程
- 如果逻辑只影响当前表单 UI，优先改行为
- 如果规则只属于某个 sheet，不要塞到全局行为
- 如果项目尚未设 key，不要继续做可回写设计

### 7.3 输出要求

智能体完成工作后，至少要交付：

1. 当前项目处于八阶段中的哪一步
2. 哪些前置条件已满足
3. 哪些缺口阻止进入使用模式
4. 改动涉及哪一层：
   - 全局行为
   - 工作表行为
   - 表单行为
   - 流程
   - 表单结构
   - 数据配置

### 7.4 禁止事项

- 禁止直接跳到测试页排查，而不先核对数据 key
- 禁止把复杂业务继续堆进按钮脚本
- 禁止把“表单行为”当成“工作表规则”的替代品
- 禁止在未定义使用模式策略前宣称项目可交付

---

## 8. 建议的页面编排顺序

结合当前产品，推荐页面顺序重组为：

1. 项目概览
2. 数据源
3. 表单设计
4. 事件与行为
5. 流程编排
6. 项目行为
7. 测试运行
8. 发布与使用模式

当前系统虽然已有统一编辑器，但从规范上应把“项目行为”和“发布/使用模式”做成更明确的独立阶段。

---

## 9. 一句话标准

一套合格的 FormFlow 项目，不是“能拖出一个表单”，而是“能从稳定数据源出发，经过明确分层的表单、事件、流程、行为和测试，最终切换到可控的使用模式并稳定运行”。
