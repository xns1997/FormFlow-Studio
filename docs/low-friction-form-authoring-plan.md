# FormFlow 低操作表单开发体验改进计划

**日期**：2026-07-14
**目标**：让普通用户从一份 Excel / CSV 到“可填写、可校验、可提交”的表单，默认不写 JS、不手工连基础流程，并在 5 分钟内完成首个可用版本。

## 1. 当前判断

当前系统已经具备 216 个节点包、27 类表单控件、24 种行为事件、17 种动作、行为模板、设计模板、流程快速连接、代码补全和项目生成 skill。能力覆盖已经足够，主要问题是能力分散：

1. 用户仍需在“数据预览 → 表单设计 → 行为定义 → 流程编排 → 测试运行”之间多次切换。
2. 表格字段、表单控件、数据绑定、校验和 CRUD 流程需要重复配置相同信息。
3. 常用业务意图被拆成很多低阶节点，用户需要理解端口、映射和执行顺序。
4. 已有 `setValues`、`requireFields`、`fillForm`、`resetForm`、`findRow(s)`、`nextSequence` 等快捷方法主要藏在代码提示中，不容易被非开发用户发现。
5. 模板解决“从哪里开始”，但还没有解决“针对我的数据自动完成到什么程度”。

因此本轮不以继续增加低阶节点数量为主，而是增加“意图级入口”，并把它们编译或展开为现有表单、行为和流程结构。

## 2. 成功指标

以“员工信息录入表单”为标准任务，改造后达到：

| 指标 | 目标 |
|---|---:|
| 上传数据到可运行表单 | ≤ 5 分钟 |
| 首次生成所需主操作 | ≤ 6 次 |
| 手工创建控件 | 默认 0 个 |
| 手工创建基础绑定 | 默认 0 个 |
| 常见 CRUD 场景手工连线 | 默认 0 条 |
| 80% 常用联动 | 无需 JS |
| 首次运行前可发现的配置错误 | ≥ 90% |
| 新用户首个表单完成率 | 提升 30% |

埋点记录：创建方式、生成耗时、手工新增控件数、手工连线数、撤销次数、首次运行错误数、发布前停留时间。

## 3. 目标体验：两条开发路径

### 3.1 快速路径：数据驱动生成

默认流程压缩为：

1. 上传 Excel / CSV。
2. 系统推断主表、主键、字段类型、枚举、必填项和只读字段。
3. 用户选择用途：录入、查询修改、审批、明细查看、统计。
4. 预览系统自动生成的表单、绑定、校验、行为和基础流程。
5. 点击“创建并运行”。

生成结果仍保存为现有 FormFlow v2 结构，用户可随时进入设计器继续精调。

### 3.2 专业路径：渐进增强

保留现有设计器和节点画布，但增加就地编辑、批量操作、自动连接、宏节点、规则 DSL 和诊断修复。简单逻辑在表单页面完成，只有跨表查询、复杂计算和外部集成才进入流程画布。

## 4. P0：先把首个表单压缩到 5 分钟（第 1～2 周）

### 4.1 “从数据生成表单”向导

新增 `FormGenerationWizard`，复用现有字段类型推断、设计模板和项目创建逻辑。

生成规则：

- string → 单行输入；长文本 → 多行输入。
- enum / 低基数字段 → 下拉选择或单选。
- number → 数字输入；date → 日期选择；boolean → 开关。
- key → 只读或自动编号；nullable=false → 必填。
- 字段数超过 12 时自动按语义和前缀分组，超过 24 时自动分页签。
- 自动创建 `dataBinding`、字段名、标签、占位符、默认值和基础校验。
- 根据用途自动生成按钮和行为：新增、查询、保存、重置、删除、提交。

向导最后展示“系统将创建什么”，允许逐项取消表单、行为或流程，避免黑盒生成。

### 4.2 表格列直接生成控件

在数据预览和表单设计之间增加三种操作：

- 单列拖入画布：创建控件并完成绑定。
- 多列拖入画布：创建自动布局的字段组并完成批量绑定。
- “为未绑定字段补齐控件”：只生成缺失部分，不覆盖现有设计。

### 4.3 一站式字段配置

在控件属性面板增加“字段任务卡”，一次配置：字段名、绑定表列、必填、只读、默认值、显示条件、校验和提交映射。属性变化自动同步到相关行为和流程参数映射，减少多处重复输入。

### 4.4 就地添加常用行为

控件右侧提供“＋联动”菜单，不离开表单设计页即可添加：

- 当值等于/包含/为空时，显示或隐藏字段。
- 启用/禁用、必填/选填其他字段。
- 计算并写入目标字段。
- 根据父字段刷新下拉选项。
- 点击按钮后保存、查询、重置、运行流程。

UI 使用自然语言规则编辑器，保存时编译为现有 `BehaviorRule`，高级用户仍可跳转到完整行为页。

### 4.5 即时诊断与一键修复

设计器顶部常驻“完成度”入口，检查：

- 可写字段未绑定。
- 必填字段没有提示或默认值。
- 按钮没有事件。
- 引用的流程、字段、节点或端口不存在。
- 主键为空/重复却启用了写回。
- 行为之间存在循环或写入冲突。

每条诊断提供“定位”和“自动修复”，不只显示错误文本。

## 5. P1：增加意图级方法与语法糖（第 3～4 周）

原则：所有语法糖都编译到现有行为动作或流程节点；项目包继续保存标准结构，不让运行时维护两套语义。

### 5.1 先把已有快捷方法产品化

将已有能力从代码提示提升为可见的“方法库”和规则构建器动作：

| 已有方法 | 产品化入口 |
|---|---|
| `setValues({...})` | 批量赋值 |
| `clearValues([...])` | 批量清空 |
| `setFieldState(name, state)` | 字段状态 |
| `requireFields([...])` | 批量必填校验 |
| `findRow` / `findRows` | 查询记录/列表 |
| `fillForm(row)` | 查询并回填 |
| `nextSequence(...)` | 自动编号 |
| `resetForm(...)` | 重置并准备下一条 |

每个方法提供参数表单、自然语言预览、示例数据试运行和“查看生成代码”。

### 5.2 新增规则 DSL

支持面向业务意图的短语法，并在保存时编译为 `Trigger / Condition / Action / SideEffect`：

```text
when 部门 == "技术部" -> show 技术栈, require 技术栈
otherwise -> hide 技术栈, clear 技术栈

compute 总价 = 数量 * 单价 on change(数量, 单价)

before submit -> require(姓名, 手机号); save employees by 员工ID

on 省份 change -> options 城市 from city_dict where 省份 = $value
```

语法编辑器需要字段/表/流程补全、中文别名、错误定位和双向自然语言预览。第一版只支持受控语法，不执行任意代码。

### 5.3 新增链式脚本糖

为脚本用户增加薄包装，底层调用现有 API：

```javascript
await ctx.fields('技术栈').show().required();
await ctx.fields(['备注', '附件']).hide().clear();
await ctx.form.require(['姓名', '手机号']).focusFirstInvalid();
await ctx.table('employees').find({ 员工ID: ctx.value('员工ID') }).fillForm();
await ctx.table('employees').upsert(ctx.form.values(), { key: '员工ID' });
await ctx.flow('employee_review').run({ employee: ctx.form.values() }).writeBack();
```

保留旧方法，链式 API 作为别名层逐步引入，避免破坏已有脚本。

### 5.4 表达式语法糖

统一表单、行为和节点属性中的字段引用：

```text
$姓名                 当前表单字段
$form.姓名            显式表单字段
$row.员工ID           当前数据行
$table.employees      数据表
$flow.result.score    流程输出
$event.value          当前事件值
@today                内置值
@user.id              当前用户
```

提供常用函数：`coalesce`、`round`、`dateDiff`、`formatDate`、`lookup`、`sum`、`unique`、`match`。表达式先解析为 AST，再由安全求值器执行，禁止用 `new Function` 承载新增语法。

## 6. P1：增加高阶节点与流程语法糖（第 3～5 周）

新增节点必须满足“替代至少 3 个现有节点或 5 次配置”的门槛。高阶节点优先实现为 macro，画布上是一个节点，调试时可以展开查看底层步骤。

### 6.1 首批 10 个高阶节点

| 节点 | 一次完成的工作 |
|---|---|
| 表单保存 `form:save` | 校验 → 映射 → insert/update/upsert → 成功提示 → 可选重置 |
| 查询并回填 `form:lookup-fill` | 按 key 查询 → 未找到处理 → 字段映射 → 回填表单 |
| 条件字段状态 `form:conditional-state` | 一条规则控制显隐、启用、必填、清空 |
| 级联选项 `form:cascade-options` | 查字典表 → 过滤 → 生成选项 → 处理默认值 |
| 派生字段 `form:computed-field` | 监听多个字段 → 计算 → 格式化 → 写回 |
| 表单完整校验 `form:validate-all` | 必填、类型、范围、正则、自定义规则汇总 |
| 主数据关联 `data:lookup-join` | 按 key 关联参考表并选择回填字段 |
| 匹配分支 `logic:match` | 替代多层 compare + switch，支持默认分支 |
| 尝试与补偿 `flow:try-catch` | 重试、超时、失败分支、补偿动作统一配置 |
| 批量映射 `data:map-fields` | 字段重命名、默认值、转换、丢弃在一张映射表完成 |

第二批根据埋点决定：审批提交、文件上传解析、通知组合、分页查询、事务写回、缓存/去抖。

### 6.2 画布操作糖

- 从端口拖到空白处：兼容节点按“常用下一步”排序，选中后自动连接并预填上下文。
- 拖节点到连线上：自动插入并重连。
- 选中多个节点：“组合为宏/子流程”。
- 复制节点时：可选择复制配置、复制并替换数据源、复制为引用。
- 连接后自动建议字段映射；名称和类型一致时自动映射。
- 节点缺少必填输入时，在节点卡片内直接补值，不必打开完整面板。
- 输入 `/` 打开命令面板：添加节点、添加配方、运行到此、查看输出、自动整理、抽取子流程。

### 6.3 流程配方

模板不只创建整个流程，还能插入可参数化片段：

- 查询 → 回填表单。
- 校验 → 保存 → 通知。
- 导入 → 清洗 → 去重 → 写入。
- 条件审批 → 通知 → 归档。
- 调 API → 映射响应 → 更新字段。

插入时只询问业务参数，如“哪张表、主键是什么、回填哪些字段”，系统负责创建节点、连线和端口映射。

## 7. P2：统一开发工作台与智能辅助（第 5～7 周）

### 7.1 统一“开发”工作台

不再要求新用户理解五个独立页面。首页按任务展示：

- 数据：已识别 18 个字段，1 个主键待确认。
- 表单：16/18 字段已绑定，可一键补齐。
- 规则：2 条联动、1 条提交规则。
- 流程：保存流程已就绪。
- 测试：3/4 场景通过。

点击任务在侧栏或抽屉内完成；专业用户仍能切换完整页面。

### 7.2 示例驱动规则生成

用户可以输入：“部门是技术部时显示技术栈；提交前姓名和手机号必填”。系统先生成结构化预览，让用户确认字段、触发器和动作，再写入行为规则。生成结果必须可视化、可撤销、可追踪，不能直接产生不可维护的长脚本。

### 7.3 自动测试样例

根据字段规则生成最小测试集：正常值、空值、边界值、错误类型、枚举外值、主键重复、联动真/假分支。发布门禁展示覆盖率，并允许“一键运行全部”。

## 8. 技术落地顺序

### 迭代 A：生成与就地规则

1. 抽取 `fieldInference` 和 `formScaffold` 纯函数。
2. 新增数据到表单生成向导。
3. 增加多列生成、自动绑定和缺失字段补齐。
4. 增加就地行为构建器和诊断 quick fix。
5. 为标准员工录入、工单创建、查询修改建立 E2E。

### 迭代 B：统一表达式与 DSL

1. 定义表达式 AST、字段引用规范和安全求值器。
2. 将 DSL 编译为现有 `BehaviorRule`。
3. 增加链式 API 兼容层和类型声明。
4. 将已有快捷方法接入可视化方法库。
5. 完成旧脚本、旧行为和旧项目兼容测试。

### 迭代 C：宏节点与配方

1. 定义 macro schema：参数、展开规则、调试映射、版本号。
2. 实现首批 5 个表单类宏节点，再实现 5 个数据/控制类宏节点。
3. 增加自动连接、插入连线、字段自动映射。
4. 建立宏展开后的执行结果与画布节点之间的调试映射。

### 迭代 D：统一工作台与质量门禁

1. 聚合数据、表单、行为、流程诊断。
2. 增加任务式工作台和完成度。
3. 自动生成测试样例和发布门禁。
4. 根据埋点淘汰低价值入口，优化默认推荐顺序。

## 9. 兼容性与设计约束

1. FormFlow v2 包结构保持不变；新增字段必须可选并带版本。
2. 语法糖保存前尽量编译为已有结构，避免只能由新版本读取。
3. 宏节点需要保留 `macroVersion` 和展开快照，保证历史项目可复现。
4. 自动生成绝不覆盖用户修改；使用稳定 ID 和差异合并，只补缺失内容。
5. 所有智能生成先预览后写入，且作为单条撤销事务。
6. 默认隐藏低频 XLSX 底层节点，但不删除；通过“高级节点”开关展示。
7. 节点新增由使用数据驱动，避免节点面板从 216 个继续无序膨胀。

## 10. 验收场景

每个迭代至少通过以下场景：

1. 员工录入：上传表格后自动生成录入表单、自动编号、必填校验、保存和重置。
2. 工单创建：问题类型联动处理组，优先级控制必填字段，提交后通知。
3. 查询修改：按编号查询、回填、编辑并按主键更新。
4. 主从数据：选择主记录后加载明细列表并编辑。
5. 外部接口：按钮调用 API，将响应映射回表单，并覆盖超时和失败路径。

验收时同时比较改造前后的点击数、页面切换数、手工连线数、脚本行数、完成时间和首次运行错误数。

## 11. 建议立即开始的最小切片

第一版只做一条闭环：

> 选择数据表 → 自动生成录入表单 → 自动绑定字段 → 自动生成“校验并保存”行为 → 预览运行。

范围控制为 8 种常见字段类型、单表、单主键、create 模式，不先做 AI、审批和跨表。这个切片能最快验证“操作数是否真的下降”，也能为后续查询修改、宏节点和 DSL 提供统一的生成基础。
