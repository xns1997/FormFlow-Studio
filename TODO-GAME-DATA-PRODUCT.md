# 游戏公司数据录入、分析、预测与可视化 TODO

> 目标：从零实现一个可录入、可分析、可预测、可视化、可回归验证的 FormFlow 游戏数据项目，作为后续官方示例的唯一参考实现。
>
> 约束：运行时业务逻辑优先使用表单规则、查询、筛选、聚合、时间序列、回归、评分和写回节点；不使用自定义 JS 节点，不使用控件内联事件脚本。

- [ ] 至少包含 5 张业务表、3 个可写录入表单、2 条分析/预测流程、1 个综合看板。
- [ ] 能完整跑通“录入事件 → 执行分析 → 执行预测 → 刷新看板”。
- [ ] 所有 Mock 数据固定 seed、固定时间范围、稳定排序，重复生成结果一致。
- [ ] 所有可编辑表均有非空、唯一、稳定的业务主键。
- [ ] 所有预测结果明确标记为模型值，不与实际统计值混淆。

## 1. 业务范围与口径

- [ ] 选择一个虚构游戏：`星港远征`。
- [ ] 数据观察窗口固定为连续 120 天。
- [ ] 玩家规模至少 1,200 人。
- [ ] 登录/关卡/关键行为事件至少 12,000 条。
- [ ] 付费订单至少 2,000 条。
- [ ] 活动配置至少 12 期，覆盖不同渠道和版本。
- [ ] 统一时区为 `Asia/Shanghai`，日期使用 `YYYY-MM-DD`，时间使用 ISO 8601。
- [ ] 在项目说明页写清指标口径、数据范围、Mock 声明和预测免责声明。

## 2. 数据模型

### 2.1 玩家档案 `players`

- [ ] 主键：`player_id`，格式 `P000001`。
- [ ] 字段：注册日期、渠道、地区、设备系统、首个版本、当前版本、年龄段、是否新玩家、生命周期状态。
- [ ] 枚举：渠道、设备系统、年龄段和生命周期状态均使用固定选项。
- [ ] 校验：注册日期不得晚于当前业务日期；版本号、渠道必填。

### 2.2 玩家事件 `player_events`

- [ ] 主键：`event_id`，格式 `EVT-YYYYMMDD-000001`。
- [ ] 字段：玩家 ID、事件时间、事件类型、会话 ID、关卡 ID、关卡结果、耗时秒数、版本、渠道、设备系统。
- [ ] 事件类型至少包含：登录、会话开始、关卡开始、关卡完成、关卡失败、活动参与、流失预警触发。
- [ ] 校验：耗时范围 `0–14400` 秒；关卡事件必须填写关卡 ID；完成/失败事件必须填写结果。

### 2.3 付费订单 `payment_orders`

- [ ] 主键：`order_id`，格式 `ORD-YYYYMMDD-000001`。
- [ ] 字段：玩家 ID、下单时间、商品类型、商品名称、金额、币种、支付状态、首充标记、渠道、版本。
- [ ] 支付状态至少包含：待支付、已支付、退款、关闭。
- [ ] 校验：金额大于 0；币种固定为 CNY；只有已支付订单进入收入分析。

### 2.4 活动配置 `campaigns`

- [ ] 主键：`campaign_id`，格式 `CMP-YYYYMM-001`。
- [ ] 字段：活动名称、活动类型、开始日期、结束日期、目标渠道、目标版本、预算、目标新增、目标收入、状态。
- [ ] 校验：结束日期不得早于开始日期；预算和目标值不得为负数。
- [ ] 状态联动：草稿可编辑，进行中锁定活动 ID，已结束不可修改日期范围。

### 2.5 日指标结果 `daily_metrics`

- [ ] 复合业务键：`metric_date + channel + version`，另存稳定主键 `metric_id`。
- [ ] 字段：DAU、MAU、D1 留存率、D7 留存率、付费人数、付费率、收入、ARPU、ARPPU、平均会话时长、关卡完成率。
- [ ] 增加 `data_kind` 字段，固定为 `actual`。
- [ ] 分析流程按业务键幂等写回，不重复插入同一天同维度结果。

### 2.6 预测结果 `metric_forecasts`

- [ ] 主键：`forecast_id`。
- [ ] 字段：生成批次、预测日期、指标名称、场景、预测值、下界、上界、训练截止日期、模型类型、数据种类。
- [ ] 场景至少包含：基准、乐观、保守。
- [ ] `data_kind` 固定为 `mock_forecast`，模型类型记录实际使用的时间序列或回归节点。
- [ ] 同一批次、日期、指标和场景只允许一条结果。

## 3. 录入表单

### 3.1 玩家事件录入

- [ ] 提供事件 ID 自动生成、玩家查询选择、事件时间、类型、关卡和耗时录入。
- [ ] 使用规则控制关卡字段显隐和必填状态。
- [ ] 保存按钮绑定“校验 → 查重 → 写入事件表 → 返回成功摘要”流程。
- [ ] 覆盖失败路径：缺玩家、重复事件 ID、耗时越界、关卡字段缺失。

### 3.2 付费订单录入

- [ ] 提供订单 ID 自动生成、玩家查询选择、商品、金额和状态录入。
- [ ] 首充标记由查询该玩家历史已支付订单的结果确定，优先使用查询与条件节点。
- [ ] 保存按钮绑定“校验 → 查重 → 写入订单表 → 更新摘要”流程。
- [ ] 覆盖失败路径：金额为零、重复订单、无效玩家、退款订单缺原订单信息。

### 3.3 活动配置录入

- [ ] 提供活动基本信息、日期范围、目标渠道/版本、预算和目标录入。
- [ ] 使用规则实现日期、预算、状态联动。
- [ ] 保存按钮绑定“校验 → 查重/更新判断 → 新增或更新 → 返回状态”流程。
- [ ] 覆盖失败路径：日期倒置、预算为负、已结束活动被修改。

## 4. 分析流程

### 4.1 每日核心指标流程

- [ ] 输入：统计日期、渠道（可选）、版本（可选）。
- [ ] 查询目标日期及回溯窗口内的玩家、事件和已支付订单。
- [ ] 计算 DAU：目标日发生有效活跃事件的去重玩家数。
- [ ] 计算 MAU：目标日前 30 天发生有效活跃事件的去重玩家数。
- [ ] 计算 D1/D7 留存：注册 cohort 在第 1/7 天再次活跃的玩家占比。
- [ ] 计算付费率：付费人数 / DAU。
- [ ] 计算 ARPU：已支付收入 / DAU。
- [ ] 计算 ARPPU：已支付收入 / 付费人数。
- [ ] 计算关卡漏斗：关卡开始人数、完成人数、失败人数和完成率。
- [ ] 按日期、渠道、版本聚合并幂等写入 `daily_metrics`。
- [ ] 对分母为 0 的指标返回 0，不产生 `NaN`、`Infinity` 或流程错误。

### 4.2 活动效果分析流程

- [ ] 输入：活动 ID。
- [ ] 查询活动周期、目标渠道和目标版本的数据。
- [ ] 计算活动参与人数、新增玩家、付费人数、收入和预算回报率。
- [ ] 对比活动前 7 天基线，输出活跃和收入提升率。
- [ ] 输出渠道贡献、版本贡献和异常订单明细。
- [ ] 无匹配事件或订单时输出空结果提示，不报错。

## 5. 预测流程

- [ ] 预测对象至少包含未来 30 天 DAU 和收入。
- [ ] 训练数据只使用 `daily_metrics.data_kind = actual` 的记录。
- [ ] 训练截止日期必须早于所有预测日期，禁止未来数据泄漏。
- [ ] 基准场景使用现有时间序列或回归节点生成中心预测。
- [ ] 乐观/保守场景使用可录入参数对基准值进行规则化调整，不使用脚本。
- [ ] 输出预测值、合理上下界和模型元数据。
- [ ] 预测结果按批次幂等写入 `metric_forecasts`。
- [ ] 训练数据不足时明确返回“样本不足”，不生成伪精确结果。
- [ ] 增加回测：保留最近 14 天作为验证集，计算 MAE 和 MAPE。
- [ ] 设定最低验收线：DAU MAPE 不高于 25%，收入 MAPE 不高于 35%；未达标仍可展示，但必须标红并提示。

## 6. 综合看板

- [ ] 顶部提供日期、渠道、版本和实际/预测场景筛选器。
- [ ] KPI 卡：DAU、MAU、D1 留存、D7 留存、付费率、收入、ARPU、ARPPU。
- [ ] 折线图：最近 30/60/90 天 DAU 与收入趋势。
- [ ] 预测图：实际值与未来 30 天三场景预测，并展示上下界。
- [ ] 留存图：D1、D7 留存趋势或 cohort 热力矩阵。
- [ ] 漏斗图：登录 → 关卡开始 → 关卡完成 → 付费。
- [ ] 柱状图：渠道收入贡献和版本活跃对比。
- [ ] 分布图：玩家付费金额分层或订单金额区间。
- [ ] 明细表：异常订单、高流失关卡、低留存渠道、预测误差超标日期。
- [ ] 页面显著标识“预测为模型结果，不代表实际经营结果”。
- [ ] 刷新按钮绑定有效分析流程；筛选为空时显示空状态，不报错。

## 7. Mock 数据设计

- [ ] 固定随机 seed，例如 `20260716`。
- [ ] 让渠道、版本、地区和设备分布具有可解释差异，不做完全均匀随机。
- [ ] 模拟周末效应、版本发布、活动拉升和节后回落。
- [ ] 模拟玩家生命周期：新用户、活跃、沉默、回流和流失。
- [ ] 支付金额使用长尾分布，并包含少量退款和关闭订单。
- [ ] 注入可被看板识别的异常：某版本关卡流失、某渠道低留存、某日收入尖峰。
- [ ] 保证跨表引用完整：事件和订单中的玩家 ID 必须存在。
- [ ] 生成后检查主键重复、空主键、孤儿引用、非法日期和越界数值。
